# Sesi 5: Menghubungkan Dataset (The Join Strategy)

**Target Sesi:** Siswa memahami logika relasional dan mampu menggabungkan dua tabel atau lebih untuk mendapatkan wawasan data yang lebih utuh menggunakan berbagai tipe JOIN.

---

## 1. Mengapa Data Harus Dipisah? (Relational Logic)

Dalam database yang baik, kita tidak menumpuk semua informasi dalam satu tabel raksasa (seperti satu _sheet_ raksasa di Excel). Kita memisahnya agar:

1. **Data lebih rapi** - Setiap tabel punya tanggung jawab satu hal saja
2. **Tidak ada duplikasi** - Informasi wilayah disimpan sekali, bukan di setiap baris responden
3. **Lebih mudah diperbarui** - Ubah satu kali, efek langsung ke semua query
4. **Konsistensi terjaga** - Tidak akan ada typo yang berbeda di baris lain

**Contoh struktur:**

- **Tabel responden:** Nama, Umur, Email, Pekerjaan, ID_Wilayah, Pendapatan, Status_Pernikahan, Tanggal_Survey
- **Tabel wilayah_ref:** ID_Wilayah, Nama_Wilayah, Region, Zona (tabel referensi)
- **Tabel kategori_pekerjaan:** ID_Pekerjaan, Nama_Pekerjaan, Sektor (tabel referensi)

**Penghubung (Key):**
Kita butuh satu kolom yang sama di kedua tabel. Biasanya disebut **Primary Key** (PK) di tabel induk dan **Foreign Key** (FK) di tabel detail.

## 3. Mengenal Jenis-Jenis JOIN

JOIN membuat kombinasi baris dari dua tabel atau lebih berdasarkan kondisi yang Anda tentukan. Ada 5 tipe utama:

---

### A. INNER JOIN (Hanya yang Berpasangan)

**Prinsip:** Hanya tampilkan baris di mana ada kecocokan di **kedua** tabel.

**Gambaran:** Dari 1000 responden, mungkin hanya 800 yang punya data lengkap di kedua tabel. Sisanya tidak akan muncul.

#### Contoh 1: Basic INNER JOIN

```sql
-- Cari nama responden dan wilayahnya berdasarkan tabel referensi
SELECT
    r.nama,
    r.wilayah,
    wr.region
FROM responden r
INNER JOIN wilayah_ref wr ON r.wilayah = wr.nama;
```

**Hasil yang diharapkan:**

- Jika responden.wilayah = 'Jakarta' dan ada row di wilayah_ref dengan nama 'Jakarta', baris ini akan muncul.
- Jika responden.wilayah = 'Kota Aneh' (tidak ada di wilayah_ref), baris ini TIDAK akan muncul.

#### Contoh 2: INNER JOIN dengan Agregasi

```sql
-- Hitung rata-rata pendapatan per sektor pekerjaan
SELECT
    kp.sektor,
    COUNT(*) AS jumlah_responden,
    ROUND(AVG(r.pendapatan), 0) AS rata_rata_pendapatan
FROM responden r
INNER JOIN kategori_pekerjaan kp ON r.pekerjaan = kp.nama
GROUP BY kp.sektor
ORDER BY rata_rata_pendapatan DESC;
```

**Penjelasan:**

- `GROUP BY kp.sektor` mengelompokkan responden berdasarkan sektor
- Hanya responden dengan pekerjaan yang ada di kategori_pekerjaan yang dimasukkan
- Hasil menunjukkan sektor apa saja yang "lengkap" datanya

#### Contoh 3: INNER JOIN Dua Kondisi

```sql
-- Cari responden yang bekerja di Teknologi dan tinggal di Jawa
SELECT
    r.nama,
    r.pekerjaan,
    wr.region,
    r.pendapatan
FROM responden r
INNER JOIN kategori_pekerjaan kp ON r.pekerjaan = kp.nama
INNER JOIN wilayah_ref wr ON r.wilayah = wr.nama
WHERE kp.sektor = 'Teknologi'
  AND wr.region = 'Jawa'
ORDER BY r.pendapatan DESC
LIMIT 10;
```

---

### B. LEFT JOIN (Prioritas Tabel Kiri)

**Prinsip:** Tampilkan **SEMUA** baris dari tabel kiri, dan tambahin data dari tabel kanan jika ada. Jika tidak ada, hasilnya `NULL`.

**Gambaran:** Ini adalah favorit para peneliti karena tidak "menghilangkan" data.

#### Contoh 1: Basic LEFT JOIN

```sql
-- Tampilkan SEMUA responden dan wilayah asal mereka (jika ada di referensi)
SELECT
    r.nama,
    r.wilayah,
    COALESCE(wr.region, 'TIDAK DITEMUKAN') AS region
FROM responden r
LEFT JOIN wilayah_ref wr ON r.wilayah = wr.nama
LIMIT 20;
```

**Hasil yang diharapkan:**

- Jakarta dengan Jawa
- NULL (atau TIDAK DITEMUKAN setelah COALESCE) jika wilayah tidak ada di referensi
- Semua 10,000 responden tetap ada di hasil

#### Contoh 2: Menemukan Data yang Hilang

```sql
-- Temukan responden yang wilayahnya TIDAK terdaftar di wilayah_ref
SELECT
    r.nama,
    r.wilayah,
    r.pendapatan
FROM responden r
LEFT JOIN wilayah_ref wr ON r.wilayah = wr.nama
WHERE wr.id IS NULL
LIMIT 20;
```

**Penjelasan:**

- `WHERE wr.id IS NULL` menangkap hanya baris di mana JOIN tidak menemukan kecocokan
- Ini adalah teknik **audit data** penting!

#### Contoh 3: LEFT JOIN dengan Agregasi dan COALESCE

```sql
-- Hitung total responden per sektor, termasuk sektor yang tidak lengkap datanya
SELECT
    COALESCE(kp.sektor, 'TIDAK TERDAFTAR') AS sektor,
    COUNT(*) AS jumlah_responden,
    ROUND(AVG(r.pendapatan), 0) AS rata_rata_pendapatan
FROM responden r
LEFT JOIN kategori_pekerjaan kp ON r.pekerjaan = kp.nama
GROUP BY kp.sektor
ORDER BY jumlah_responden DESC;
```

---

### C. RIGHT JOIN (Prioritas Tabel Kanan)

**Prinsip:** Seperti LEFT JOIN, tapi yang dipertahankan adalah tabel **kanan**.

**Catatan:** Jarang digunakan karena `RIGHT JOIN` bisa ditulis sebagai `LEFT JOIN` dengan menukar posisi tabel.

#### Contoh: RIGHT JOIN Sederhana

```sql
-- Tampilkan SEMUA kategori pekerjaan, dan berapa banyak responden di setiap kategori
SELECT
    kp.nama,
    kp.sektor,
    COUNT(r.id) AS jumlah_responden
FROM responden r
RIGHT JOIN kategori_pekerjaan kp ON r.pekerjaan = kp.nama
GROUP BY kp.id, kp.nama, kp.sektor
ORDER BY jumlah_responden DESC;
```

**Penjelasan:**

- Semua baris di kategori_pekerjaan akan muncul, bahkan jika tidak ada responden
- Ini berguna untuk "master data validation"

---

### D. FULL OUTER JOIN (Gabung Semua)

**Prinsip:** Tampilkan **SEMUA** baris dari kedua tabel. Jika tidak ada kecocokan, salah satu sisi akan `NULL`.

**Catatan:** Jarang dipakai dalam praktik, tapi berguna untuk deteksi duplikasi atau inconsistency.

```sql
-- Deteksi inconsistency antara responden dan kategori pekerjaan
SELECT
    r.id,
    r.pekerjaan,
    kp.nama AS kategori_pekerjaan_terdaftar,
    CASE
        WHEN r.pekerjaan IS NULL THEN 'Hanya di Kategori'
        WHEN kp.nama IS NULL THEN 'Hanya di Responden'
        ELSE 'Match'
    END AS status
FROM responden r
FULL OUTER JOIN kategori_pekerjaan kp ON r.pekerjaan = kp.nama
WHERE r.pekerjaan IS NULL OR kp.nama IS NULL
LIMIT 20;
```

---

### E. CROSS JOIN (Perkalian Cartesian - Hati-hati!)

**Prinsip:** Buat kombinasi SETIAP baris tabel A dengan SETIAP baris tabel B.

**Peringatan:** Jika A punya 1000 baris dan B punya 100 baris, hasil akan 100,000 baris! Gunakan dengan hati-hati.

```sql
-- CONTOH: Buat semua kombinasi wilayah x sektor (1000 kombinasi)
-- JANGAN JALANKAN TANPA LIMIT!
SELECT
    wr.nama AS wilayah,
    kp.sektor
FROM wilayah_ref wr
CROSS JOIN kategori_pekerjaan kp
LIMIT 20;
```

## 4. Menggabungkan Lebih dari Dua Tabel (Multi-Join)

Seringkali analisis memerlukan data dari 3+ tabel. SQL memungkinkan "VLOOKUP berantai" dengan efisien.

#### Contoh 1: Tiga Tabel, Dua Kondisi

```sql
-- Skenario: Ambil responden lengkap dengan wilayah dan sektor pekerjaan
SELECT
    r.id,
    r.nama,
    r.pendapatan,
    wr.region,
    kp.sektor,
    kp.tier_gaji
FROM responden r
LEFT JOIN wilayah_ref wr ON r.wilayah = wr.nama
LEFT JOIN kategori_pekerjaan kp ON r.pekerjaan = kp.nama
WHERE r.pendapatan > 10000000
ORDER BY r.pendapatan DESC
LIMIT 20;
```

**Penjelasan:**

- Urutan JOIN tidak masalah, tapi `LEFT JOIN` memastikan semua responden tetap ada
- Filter `WHERE` diterapkan **setelah** semua JOIN selesai

#### Contoh 2: Agregasi Tiga Tabel

```sql
-- Tampilkan rata-rata pendapatan per kombinasi (wilayah + sektor)
SELECT
    wr.region,
    kp.sektor,
    COUNT(*) AS jumlah_responden,
    ROUND(AVG(r.pendapatan), 0) AS rata_rata_pendapatan
FROM responden r
LEFT JOIN wilayah_ref wr ON r.wilayah = wr.nama
LEFT JOIN kategori_pekerjaan kp ON r.pekerjaan = kp.nama
GROUP BY wr.region, kp.sektor
HAVING COUNT(*) >= 3
ORDER BY rata_rata_pendapatan DESC;
```

## 5. Best Practice: Menghindari "Data Bengkak" (Data Duplication)

Salah satu kesalahan pemula adalah hasil query menjadi jauh lebih banyak dari jumlah baris aslinya.

#### Penyebab: Duplikasi ID

Bayangkan:

- Tabel responden: 1,000 baris unique responden
- Tabel hasil_tes: 3,000 baris (satu responden bisa ikut tes 3 kali)

Jika Anda JOIN tanpa hati-hati, hasilnya akan 3,000 baris! Terlihat seperti masalah, padahal itu mungkin benar (jika responden memang ikut tes 3 kali).

#### Strategi Deteksi dan Solusi

```sql
-- LANGKAH 1: Cek jumlah baris responden
SELECT COUNT(*) AS total_responden FROM responden;
-- Result: 10,000

-- LANGKAH 2: Cek jumlah unik pekerjaan (kategori)
SELECT COUNT(DISTINCT pekerjaan) AS unique_pekerjaan FROM responden;
-- Result: 25

-- LANGKAH 3: JOIN dan cek hasilnya
SELECT COUNT(*) AS hasil_after_join
FROM responden r
LEFT JOIN kategori_pekerjaan kp ON r.pekerjaan = kp.nama;
-- Jika LEBIH dari 10,000: Ada duplikasi di kategori_pekerjaan!
-- Jika NORMAL: Hanya data yang tidak match

-- LANGKAH 4: Audit mana yang tidak match
SELECT
    r.pekerjaan,
    COUNT(*) AS jumlah_responden,
    CASE WHEN kp.id IS NULL THEN 'TIDAK MATCH' ELSE 'MATCH' END AS status
FROM responden r
LEFT JOIN kategori_pekerjaan kp ON r.pekerjaan = kp.nama
GROUP BY r.pekerjaan, kp.id
ORDER BY status DESC;
```

**Kesimpulannya:** Selalu verifikasi jumlah baris sebelum dan sesudah JOIN. Jika ada lonjakan yang tidak sesuai harapan, investigasi kolom kunci (ID/FK) untuk menemukan duplikasi.

## 6. Tabel Perbandingan JOIN untuk Referensi

| Jenis Join     | Tabel Kiri | Tabel Kanan | Use Case                             | Stata Equivalent        |
| :------------- | :--------- | :---------- | :----------------------------------- | :---------------------- |
| **INNER JOIN** | Match saja | Match saja  | Hanya data lengkap di kedua tabel    | `merge 1:1, nogenerate` |
| **LEFT JOIN**  | Semua      | Match saja  | Master data dari kiri, cek duplikasi | `merge 1:1`             |
| **RIGHT JOIN** | Match saja | Semua       | Validasi master data di kanan        | `merge 1:1, _all`       |
| **FULL OUTER** | Semua      | Semua       | Deteksi inconsistency                | `merge 1:1, _all`       |
| **CROSS JOIN** | Semua      | Semua       | Buat semua kombinasi (α × β)         | `expand` / `fillin`     |

## 7. Teknik Alias dan Readability

Ketika bekerja dengan beberapa tabel, alias membuat query lebih mudah dibaca.

```sql
-- Buruk (terlalu verbose)
SELECT responden.nama, responden.pendapatan, wilayah_ref.region
FROM responden
JOIN wilayah_ref ON responden.wilayah = wilayah_ref.nama;

-- Baik (clean dan mudah dibaca)
SELECT r.nama, r.pendapatan, w.region
FROM responden r
JOIN wilayah_ref w ON r.wilayah = w.nama;
```

**Konvensi:**

- Gunakan huruf kecil untuk alias awal tabel: `r`, `w`, `kp`, dst.
- Deklarasikan di klausa `FROM`: `FROM responden r`
- Konsisten di seluruh query

## Tips & Catatan Penting

**1. Kapan Pakai INNER vs LEFT?**

- **INNER:** Hanya kalau data kedua tabel PASTI match (misal: join dengan PK yang tidak bisa NULL)
- **LEFT:** Default choice untuk peneliti. Jaga semua data, deteksi mismatch dengan `IS NULL`

**2. Jangan Lupa Alias!**

```sql
-- SALAH (membingungkan mana kolom dari mana)
SELECT id, nama, wilayah FROM responden JOIN wilayah_ref ...

-- BENAR
SELECT r.id, r.nama, w.wilayah FROM responden r JOIN wilayah_ref w ...
```

**3. JOIN Condition Order Matters**

```sql
-- Baik: Filter setelah JOIN lengkap
SELECT ... FROM a JOIN b ON ... WHERE condition;

-- Risiko: Filter dalam ON bisa jadi INNER JOIN secara implicit
SELECT ... FROM a LEFT JOIN b ON ... AND filter_condition;
-- Gunakan WHERE, bukan AND, untuk safety
```

**4. Bayangkan JOIN seperti VLOOKUP Berantai**

- VLOOKUP di Excel: Coba cocokkan satu value, ambil hasilnya
- SQL JOIN: Coba cocokkan, jika match ambil semua kolom, jika tidak match (LEFT JOIN) biarkan NULL

**5. Common Mistakes**

- Lupa `ON` condition → CROSS JOIN tidak disengaja (jutaan baris!)
- Pakai `WHERE` di INNER JOIN untuk safety → Gunakan saja karena hasilnya sama
- Assumption duplikasi tidak ada → Selalu verifikasi dengan `COUNT(DISTINCT id)`

## Tabel Referensi: Kapan Gunakan Jenis JOIN

| Pertanyaan                                                        | Jawaban                    | JOIN Type  |
| :---------------------------------------------------------------- | :------------------------- | :--------- |
| "Berapa responden yang punya data lengkap di KEDUA tabel?"        | INNER JOIN                 | INNER      |
| "Berapa responden TOTAL, tapi tunjukkan data pelengkap jika ada?" | LEFT JOIN                  | LEFT       |
| "Apakah semua kategori pekerjaan punya responden?"                | RIGHT JOIN atau sub-query  | RIGHT      |
| "Apa saja yang tidak match di kedua tabel?"                       | FULL OUTER + WHERE IS NULL | FULL OUTER |
| "Buat semua kombinasi kategori X wilayah"                         | CROSS JOIN (hati-hati!)    | CROSS      |
